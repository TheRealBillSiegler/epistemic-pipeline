# Dogfood experiment — the repo's own review history as an epistemic ledger

**Date:** 2026-07-12
**Question:** Does a belief ledger over a project's claims answer questions the GitHub issue list cannot? This tests the reframe "the epistemic pipeline is the belief layer of intent-driven development" — cheaply, before any design work.
**Method:** `run.py` encodes nine real evidence events from this project's 2026-07-11/12 review history (five blind-spot lenses, two direct code verifications, one measurement, one deep-research pass) as ratings over eight real claims, through the production `ingest_rating` path. Roots are evidence channels. Output: `output.md`, regenerated by `uv run python -X utf8 run.py`.

## What the ledger answered that the issue list cannot

**1. Graded verification status (Q1).** The tracker shows eight items, all equally "Open." The ledger separates them: #31 and #33 are the best-evidenced findings (P 0.83 / 0.79, three distinct roots each); the research-as-avoidance claim and the #41 novelty verdict sit at the single-root settledness floor (0.50). Order emerged from evidence structure, not from anyone's recollection.

**2. Attention allocation (Q2).** The single-root query flagged two claims — including the #41 novelty verdict, which *felt* settled (adversarially verified, 105 agents) but is honestly one evidence channel: one harness run. The ledger surfaced an under-verification the author of the evidence had not noticed. That is the instrument doing its job on its own operator.

**3. Contested-but-examined, distinguished from unknown (Q3).** The corrected #26 deferral claim lands at P = 0.52 with settledness 0.67 — "well-examined and genuinely unresolved," which is exactly right: strong original support (0.9), strong later disconfirmation (0.15). On the issue tracker this state exists only as a comment a reader must find and weigh. The vacuous-vs-conflicted distinction did real work on real history.

**4. Retraction in one filter (Q4).** Discrediting the original n=34 interpretation and re-folding dropped the claim from P 0.52 to 0.33 — the #42 mechanism demonstrated live, with no new machinery, because beliefs are a pure fold.

**5. Rater-correlation, quantified on our own data (Q5).** Treating the five same-model lenses as one root instead of five: #33 falls 0.75 → 0.50; #31 and #30 barely move because their settledness comes from genuinely independent channels (code reading, pytest). The #35 wall stopped being an argument and became a number.

## Honest limitations

- **The confidence inputs were authored by the experimenter.** The arithmetic is honest; the inputs are judgment. In the IDD framing many inputs become mechanical (a test run is not an opinion), which is the point — but this run does not prove that.
- **Claim identity worked because the strings were hand-controlled** — the IDD stable-ID condition, simulated. This validates that the domain *can* dissolve #33; it says nothing about the free-text case.
- **n = 8 claims, 9 events.** A demonstration of expressiveness, not a measurement of usefulness. The real test is weeks of use against live work.

## Verdict

The framing earns a design pass. The ledger produced three outputs the issue list structurally cannot — graded verification with root counts, quantified sensitivity to rater independence, one-filter retraction — and surfaced one genuinely new fact about this project's own epistemic state (the novelty verdict is single-rooted; it should get a second, different-channel check before anything leans on it).

Follow-up decisions this does *not* make: whether to pivot the product framing (Tier 2 — needs the dev-tooling prior-art pass this record explicitly did not run), and where the ledger-over-IDD integration would live.
